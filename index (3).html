ToneLab

Spectrally Controlled Parallel Multi-FX Plugin · JUCE / C++17 · VST3 · AU

ToneLab is an audio effects plugin that decouples what effect is applied from where in the frequency spectrum it acts. Rather than processing a signal as a monolithic whole, ToneLab splits audio into five independent spectral bands via a Linkwitz-Riley crossover filter bank, routes each band through its own dedicated effects chain, and recombines the results in a phase-coherent parallel architecture.

The project was developed as a capstone for ATLAS 4010, exploring the intersection of real-time DSP constraints, JUCE plugin architecture, and interaction design for audio tooling. The core engineering challenge was building a parallel processing architecture that satisfies the hard real-time requirements of the audio thread: zero allocation, no locks, and sample-accurate parameter automation, while remaining musically useful.

01 — Motivation

Why build this?
The problem space.

Background
Frequency-aware effects processing

Standard insert effects chains apply processing to the full audio spectrum. This is acoustically appropriate for some effects (a compressor shaping dynamics, for instance) but creates unwanted interactions for others. A reverb applied to a full mix introduces low-frequency wash. A distortion applied to a vocal bus blends harmonic saturation of the fundamental frequencies with the high-frequency content in ways that are often undesirable.

The conventional workaround is a parallel send architecture: duplicate the signal to multiple buses, insert an EQ on each to isolate a frequency range, apply the desired effect, and sum back to the main channel. This works, but requires significant manual routing overhead: a separate bus per effect, careful gain staging, and session complexity that grows with each added process.

ToneLab asks: what if the frequency-aware routing was native to the plugin itself?

Design Goal
Collapse the routing into a single insert

The primary design goal was to internalize the parallel send architecture into a single VST3/AU insert that behaves like any other plugin in a DAW: five independent spectral bands, each with its own effect chain, no additional routing, no auxiliary buses. Insert ToneLab, configure the bands, mix the result.

The secondary goal was to do this without violating real-time audio constraints: all processing must complete deterministically within the audio buffer period, with no heap allocation on the audio thread and no blocking operations that could cause dropouts.

Scope & Constraints
Academic context and technical boundaries

ToneLab was developed within the ATLAS 4010 capstone framework over one academic year at CU Boulder. The project was scoped as a solo engineering effort, focusing on DSP architecture and plugin development rather than visual design polish or feature breadth. The five effect types (Distortion, Chorus, Reverb, Delay, Saturation) were chosen for their range of spectral interaction characteristics, not to be exhaustive.

The implementation deliberately avoids third-party DSP libraries beyond JUCE's built-in dsp:: module, keeping the codebase self-contained and making the signal processing logic legible to anyone reviewing the source.

02 — Architecture

Signal flow &
module design.

Stage Module Responsibility Notes
1 — InputAudio buffer from host — Raw PCM from the DAW's audio engine. Stereo, 32-bit float. Buffer size and sample rate negotiated at prepareToPlay()
2 — SplitCrossover filter bank SpectrumSplitter Divides incoming audio into five frequency bands using cascaded Linkwitz-Riley filters. Each band is copied into a pre-allocated band buffer. LR4 (4th-order) filters chosen for flat summed magnitude response at crossover points
3 — ProcessPer-band FX chains LaneProcessor[5] Each of the five lanes applies its assigned effect independently to its frequency-isolated buffer. Lanes run concurrently and do not share state. All lanes read from their band buffer; write to the same buffer in place
4 — RecombineParallel mixer ParallelMixer Sums the five processed band buffers back into the main output buffer using per-lane gain values sourced from the APVTS parameter tree. Dry signal is preserved as a separate reference; wet/dry is applied at master output
5 — OutputStereo out — Processed stereo buffer returned to the DAW host. No latency compensation required. LR4 filters introduce phase shift, not time delay
03 — Design Decisions

Key technical choices
and their rationale.

Filter Design
Why Linkwitz-Riley over Butterworth?
A naive crossover using matched Butterworth filters causes a 3 dB peak at the crossover frequency when bands are summed. Linkwitz-Riley (LR) filters are designed as the square of a Butterworth — a matched LR pair sums to exactly 0 dB at all frequencies. This is a prerequisite for a parallel architecture where unchanged output when all lane gains are equal is a correctness requirement, not a nice-to-have.
Concurrency
APVTS over manual atomic parameters
JUCE's AudioProcessorValueTreeState provides a managed parameter tree with thread-safe atomic reads on the audio thread, built-in DAW automation support, and serialisation to XML for session recall. The alternative of managing raw std::atomic<float> parameters manually would have required reimplementing all of this infrastructure. APVTS was the correct abstraction for the scope of this project.
Memory
Pre-allocating band buffers
The audio thread in a real-time context must never allocate or deallocate heap memory. new, delete, and std::vector::push_back() that triggers reallocation are all prohibited. ToneLab allocates five juce::AudioBuffer<float> instances in prepareToPlay(), which is called on a non-real-time thread, and reuses them for the lifetime of the plugin session.
Parallelism
Sequential lanes vs. multi-threaded processing
An early design explored running the five lane processors on a thread pool. This was abandoned for two reasons: the overhead of thread synchronisation at the short buffer sizes used in practice (128 to 256 samples) exceeds the compute savings; and managing thread safety across five lanes sharing APVTS parameter state added significant complexity for marginal gain at the scales this plugin operates at.
04 — Implementation

Core code,
annotated.

PluginProcessor.cpp — processBlock() Audio thread · zero allocation path 
// processBlock is called by the DAW on the audio thread.
// Hard rule: no heap allocation, no blocking, no exceptions.

void ToneLabProcessor::processBlock(
    juce::AudioBuffer<float>& buffer,
    juce::MidiBuffer& midiMessages)
{
    juce::ScopedNoDenormals noDenormals;  // prevent denormal performance traps

    for (int lane = 0; lane < NUM_LANES; ++lane)
    {
        // Copy input into pre-allocated band buffer (no new allocation)
        bandBuffers[lane].makeCopyOf(buffer);

        // Apply crossover filter for this band's frequency range
        splitter.processForBand(bandBuffers[lane], lane);

        // Apply this lane's effect chain (distortion, reverb, etc.)
        laneProcessors[lane].process(bandBuffers[lane]);
    }

    // Sum all five processed bands back into the output buffer
    mixer.recombine(bandBuffers, buffer, laneGains);
}
SpectrumSplitter.h — crossover filter bank LR4 · phase-coherent band splitting 
// Linkwitz-Riley 4th-order crossover bank.
// Flat magnitude response when all bands are summed.

class SpectrumSplitter
{
public:
    void prepare(const juce::dsp::ProcessSpec& spec);
    void setCrossoverFrequency(int crossoverIndex, float hz);
    void processForBand(juce::AudioBuffer<float>& buffer, int bandIndex);

private:
    // NUM_CROSSOVERS = NUM_LANES - 1 = 4 filter pairs
    std::array<juce::dsp::LinkwitzRileyFilter<float>, NUM_CROSSOVERS> lpFilters;
    std::array<juce::dsp::LinkwitzRileyFilter<float>, NUM_CROSSOVERS> hpFilters;
};
// Note 01
ScopedNoDenormals
Denormal floating-point numbers (values very close to zero) cause catastrophic CPU slowdowns on x86 processors. ScopedNoDenormals sets the FTZ/DAZ CPU flags for the scope of processBlock(), flushing denormals to zero without branching logic in the DSP code.
// Note 02
makeCopyOf vs addFrom
makeCopyOf is used during the split stage to duplicate the dry input into each band buffer independently. addFrom is used in ParallelMixer::recombine() to accumulate bands into the output. Both operate on pre-allocated buffers with no allocation on the hot path.
// Note 03
laneGains from APVTS
Per-lane gain values are read from the APVTS parameter tree using getRawParameterValue(), which returns an std::atomic<float>*. The value is loaded once at the start of processBlock() and cached as a local, avoiding repeated atomic loads inside the inner sample loop.
05 — Dev Environment

Getting it running
on my machine.

01
Machine
MacBook Pro 15-inch, 2019. 2.4 GHz 8-Core Intel Core i9, 32 GB 2400 MHz DDR4, Radeon Pro Vega 16. Running macOS Sequoia 15.7.4.
02
Build toolchain
The project was configured in Projucer and built through Xcode. Projucer handles the JUCE module linking and generates the Xcode project; from there the plugin was built and run locally as a VST3 and AU without any additional packaging or distribution step.
03
DAW testing
All testing was done in Logic Pro. After each build, the AU component was copied to ~/Library/Audio/Plug-Ins/Components/ and validated with auval before rescanning in Logic. Logic's built-in AU validation gave useful error output when the plugin metadata or parameter tree was malformed.
04
Windows build
A Windows version was built later in development. The same Projucer project was used to generate a Visual Studio solution, which required adjusting a few platform-specific JUCE component flags. Cross-platform differences were minor but real, mostly around font rendering and component sizing in the editor.
